# บันทึกการตัดสินใจทางสถาปัตยกรรม (Decision Log)

หน้านี้รวบรวมการตัดสินใจหลักๆ ทางด้านเทคโนโลยีและการออกแบบของโปรเจค **Invest** พร้อมระบุเหตุผลและทางเลือกอื่นที่เคยพิจารณา

---

## 1. การเลือกใช้ Google Sheets และ Google Apps Script เป็น Backend
- **การตัดสินใจ**: ใช้ Google Sheets ในการบันทึกข้อมูลหลัก และเขียน API ครอบด้วย Google Apps Script
- **บริบท / เหตุผล**:
  - **ฟรี 100%**: ไม่มีค่าบำรุงรักษาฐานข้อมูลและการโฮสต์เซิร์ฟเวอร์ใดๆ
  - **GUI ที่ง่าย**: เจ้าของพอร์ตสามารถเข้าไปตรวจสอบ ปรับปรุง หรือป้อนข้อมูลผ่านโปรแกรม Google Sheets บนคอมพิวเตอร์หรือโทรศัพท์มือถือโดยตรงได้สะดวกรวดเร็ว
  - **ความปลอดภัย**: ทำงานภายใต้ระบบคลาวด์ของบัญชี Google ส่วนบุคคล
- **ทางเลือกอื่นที่พิจารณา**: Firebase, Supabase
  - *เหตุผลที่ไม่เลือก*: มีขั้นตอนการคอนฟิกเพิ่มขึ้น มีความปลอดภัยสูงขึ้นแต่อาจสร้างภาระให้ผู้ใช้ในการสร้าง GUI ตารางข้อมูลเพื่อแก้ไขด้วยมือภายนอกเว็บแอพ

---

## 2. การจัดการสไตล์ด้วย Vanilla CSS (ไม่ใช่ Tailwind หรือ CSS-in-JS)
- **การตัดสินใจ**: เขียน CSS สไตล์มาตรฐานระดับต่ำในไฟล์ `src/index.css`
- **บริบท / เหตุผล**:
  - **ประสิทธิภาพและความยืดหยุ่นสูงสุด**: ไม่ต้องรันกระบวนการคอมไพล์ CSS ขนาดใหญ่ การเขียน CSS ดิบให้พลังสูงสุดในการควบคุมคุณสมบัติกระจกโปร่งแสง (`backdrop-filter`) การทำ Gradient ไล่เฉด และการเคลื่อนไหวที่ซับซ้อน
  - **ไม่มีขั้นตอนยุ่งยาก**: เหมาะสำหรับแอพพลิเคชันขนาดกระทัดรัดที่ต้องการความรวดเร็วและใช้ Component พื้นฐานของระบบในการปรับแต่งโครงร่าง
- **ทางเลือกอื่นที่พิจารณา**: TailwindCSS
  - *เหตุผลที่ไม่เลือก*: มีการขัดจังหวะการจัดแต่งสไตล์ในโค้ด React (Class ยยาวเหยียด) และต้องมีไฟล์คอนฟิก `tailwind.config.js` เพิ่มเติม

---

## 3. การเลือกไม่ใช้ระบบจัดการสถานะภายนอก (เช่น Redux, Zustand)
- **การตัดสินใจ**: ควบคุมและบริหารจัดการ State ทั้งหมดผ่าน React Hooks พื้นฐาน (`useState`, `useMemo`, `useRef`) ในคอมโพเนนต์สูงสุด (`App`)
- **บริบท / เหตุผล**:
  - **ความซับซ้อนต่ำ**: การส่งผ่านข้อมูล (Prop Drilling) เกิดขึ้นเพียงแค่ 1 ชั้นไปยังคอมโพเนนต์ `StockCard` และ `UpdateModal` การใช้ระบบอื่นภายนอกจะทำให้โค้ดซับซ้อนเกินความจำเป็น (Overengineering)
  - **ความเสถียรสูง**: โครงสร้างแบบ React State มาตรฐานทำงานได้รวดเร็วและประมวลผลข้อมูลใหม่ทันทีที่มีการเปลี่ยนแปลง
- **ทางเลือกอื่นที่พิจารณา**: Context API, Zustand
  - *เหตุผลที่ไม่เลือก*: ไม่จำเป็นเนื่องจากลักษณะของข้อมูลมีทิศทางไหลลงล่างชั้นเดียวที่ชัดเจนมาก

---

## 4. วิธีการแก้ปัญหา CORS ในการยิงส่งข้อมูลด้วย `Content-Type: text/plain`
- **การตัดสินใจ**: ใช้การตั้งค่า Fetch ส่ง Payload JSON ไปยัง Google Apps Script โดยใช้หัวข้อ `'Content-Type': 'text/plain'`
- **บริบท / เหตุผล**:
  - **ข้อจำกัดของ Google Apps Script**: การส่งแบบ `'application/json'` มาตรฐานจะบีบบังคับให้บราว์เซอร์ส่ง CORS Preflight Request (`OPTIONS` method) ซึ่งเซิร์ฟเวอร์ของ Google Apps Script มักจะไม่ตอบรับหรือส่งกลับค่า Redirect ทำให้การสื่อสารล้มเหลว
  - **ทางออกที่เรียบง่าย**: การระบุเป็น `text/plain` บราว์เซอร์จะอนุญาตให้ส่งคำขอไปได้ทันทีโดยมองเป็น Simple Request และฝั่งเซิร์ฟเวอร์จะรับข้อความดิบมาแปลงเป็นออบเจกต์เพื่อนำไปเขียนลงชีทโดยไม่มีปัญหาด้าน CORS
- **ทางเลือกอื่นที่พิจารณา**: การผ่าน Reverse Proxy หรือการใช้ JSONP
  - *เหตุผลที่ไม่เลือก*: ซับซ้อนและมีค่าใช้จ่ายเพิ่มเติม

---

## 5. การพัฒนาเครื่องคิดเลขในตัวแบบคีย์บอร์ด Popover (Embedded Popover Calculator)
- **การตัดสินใจ**: ออกแบบเครื่องคิดเลขจำลองให้ลอยอยู่บนช่องอินพุตตัวเลขใน Modal อัปเดตเมื่อคลิกไอคอนเครื่องคิดเลข
- **บริบท / เหตุผล**:
  - **จุดเจ็บปวดของผู้ใช้ (User Pain Point)**: บ่อยครั้งการกรอกยอดซื้อขายแต่ละครั้ง ผู้ใช้ต้องเอาตัวเลขหลายๆ รายการมาบวกกัน (เช่น ซื้อครั้งที่หนึ่ง $50 ซื้อครั้งที่สอง $60) การสลับไปเปิดโปรแกรมเครื่องคิดเลขภายนอกหน้าเว็บสร้างความน่าหงุดหงิด
  - **สร้างประสบการณ์ใช้งานระดับพรีเมียม (Premium UX)**: การฝังเครื่องคิดเลขไว้ในตัวบวกกับการแปลงสมการแบบสดๆ (Live preview of mathematical expressions) ช่วยอำนวยความสะดวกอย่างล้นหลาม โดยที่ยังคงความสวยงามสไตล์ Glassmorphic ของธีมแอพพลิเคชันเอาไว้
